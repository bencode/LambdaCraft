from collections import Counter

def bm25(corpus, query, k1=1.2, b=0.75):
    N = len(corpus)
    avgdl = sum(len(doc) for doc in corpus) / N
    df = Counter()
    for doc in corpus:
        for term in set(doc):
            df[term] += 1

    scores = []
    for doc in corpus:
        tf = Counter(doc)
        dl = len(doc)
        score = 0.0
        for term in query:
            n = df.get(term, 0)
            if n == 0:
                continue
            idf = log(1 + (N - n + 0.5) / (n + 0.5))
            num = tf[term] * (k1 + 1)
            den = tf[term] + k1 * (1 - b + b * dl / avgdl)
            score += idf * num / den
        scores.append(score)
    return scores

corpus = [
    "bm25 parameters guide".split(),
    "bm25 bm25 bm25 tutorial".split(),
    "okapi k1 b tuning explained bm25".split(),
]
query = ["bm25", "parameters"]

for doc, s in zip(corpus, bm25(corpus, query)):
    print(f"{s:.4f}  {' '.join(doc)}")`

BM25 是一个排序打分算法：给它一批文档和一个查询词，它为每篇文档算出一个相关度分数，分数高的排在前面 —— 这个分数就是搜索引擎或 RAG 候选召回的排序依据。几乎所有搜索引擎和 RAG 系统的关键词检索底下都跑它。

[BM25 总览](/ir-rag/bm25) 给出了完整公式和三层直觉（稀有度 / 饱和 / 长度归一化）。这篇按 idf、饱和、长度归一化三项拆开，对每一项指出它在代码里的位置，再说明它为什么长成这个形式。

## 最小示例

BM25 的整体骨架是这样：

```python
for t in query:
    idf  = rarity(t)                  # 第 3 节：idf 项
    sat  = saturation(tf, k1)         # 第 4 节：饱和项
    norm = length_norm(dl, avgdl, b)  # 第 5 节：长度项
    score += idf * sat / norm
```

三个部件彼此独立。下面三节把 `rarity / saturation / length_norm` 各自拆回 BM25 公式里对应的那项，最后一节再合起来得到可运行的 Python 实现。

## BM25 公式

$$
\text{score}(d, q) = \sum_{t \in q} \underbrace{\text{idf}(t)}_{\text{稀有度}} \cdot \underbrace{\frac{\text{tf}(t, d) \cdot (k_1 + 1)}{\text{tf}(t, d) + k_1 \cdot \left(1 - b + b \cdot \frac{|d|}{\text{avgdl}}\right)}}_{\text{饱和 tf（内含长度归一化）}}
$$

这是 per-term 打分再求和：对 query 里的每个词 $t$ 算一个贡献值，把贡献值累加得到文档分。单项贡献 = idf 乘以饱和 tf，而文档长度归一化嵌在饱和项的分母里，不独立出现。三个部件彼此独立，下面一节一项单独拆：

- **idf 项**：词在 collection 里有多稀有，只和 `N`、`df` 有关
- **饱和项**：同一个词多次出现贡献边际递减，由 $k_1$ 控制
- **长度项**：长文档整体打折，嵌在饱和项分母里，由 $b$ 控制

去掉 `1 +` 那个偏移（Lucene / `rank_bm25` 加的防负数保护），剩下的 $\log\frac{N-n+0.5}{n+0.5}$ 是 **RSJ weight 在"无 relevance 信息"假设下的自然退化**。下面把这条推导走完。

### PRP

> If retrieved documents are ordered by decreasing probability of relevance on the data available, then the system's effectiveness is the best that can be obtained for the data.

按"相关概率"降序排文档，在单用户 / 单 query 下是期望损失最小的策略（Bayes Optimal Decision Rule）。后面所有推导都只盯一件事：**估每篇文档的相关概率**。

### BIM · 从整篇概率到 per-term 求和

PRP 让我们按 $P(R=1 \mid d, q)$ 降序排文档，但这个概率直接估不了 —— 文档空间太大，标注数据太薄，根本没法对某个具体的 $(d, q)$ 组合给出可靠估计。

**Binary Independence Model** 做三个简化让它能估：

1. 文档记成二值词出现向量（出现 / 不出现），丢掉词频
2. 给定 $R$ 时，各词的出现彼此独立
3. 只看 query 里出现过的词，query 外的词假设和 relevance 无关

有了这三条，整篇相关概率就能写成每个命中词的 log odds ratio 之和。

下面 6 步每一步都是**保序变换** —— 只改公式形式，不改任何两个文档之间的相对排序。严格说：若变换前文档 $a$ 的分数低于 $b$，变换后仍然低于 $b$。高等数学里两个最常见的保序操作是：

- 单调递增函数（比如 $\log$）：$x < y$ 等价于 $\log x < \log y$，所以取 log 前后排名不变
- 乘以或加上一个对所有候选文档相同的常数：整排平移不改相对位置

这 6 步就用这两类操作，把 $P(R=1 \mid \vec{x}, \vec{q})$ 逐步化成只含 query 命中词的 per-term 求和。

**记号**：

- $\vec{x} = (x_1, ..., x_M)$：文档的二值词出现向量，$x_t = 1$ 表示词 $t$ 在文档里出现过
- $\vec{q}$：query 的同类二值向量
- $\propto_q$：query 固定下的保序等价，两侧对所有候选文档排序相同

$$
\begin{aligned}
P(R=1 \mid \vec{x}, \vec{q})
&\propto_q O(R \mid \vec{x}, \vec{q})
&& (1)\ \text{取 odds} \\[2pt]
&\propto_q \frac{P(\vec{x} \mid R=1, \vec{q})}{P(\vec{x} \mid R=0, \vec{q})}
&& (2)\ \text{Bayes + 丢查询常数} \\[2pt]
&= \prod_{t=1}^{M} \frac{P(x_t \mid R=1, \vec{q})}{P(x_t \mid R=0, \vec{q})}
&& (3)\ \text{条件独立假设} \\[2pt]
&= \prod_{t: x_t=1} \frac{p_t}{u_t} \cdot \prod_{t: x_t=0} \frac{1-p_t}{1-u_t}
&& (4)\ \text{按 present/absent 拆} \\[2pt]
&\propto_q \prod_{t: x_t=q_t=1} \frac{p_t(1-u_t)}{u_t(1-p_t)}
&& (5)\ \text{只留 query 词 + 代数重排} \\[2pt]
&\propto_q \sum_{t: x_t=q_t=1} \log \frac{p_t(1-u_t)}{u_t(1-p_t)} = \text{RSV}_d
&& (6)\ \text{取 log}
\end{aligned}
$$

每一步：

- **(1)** odds 和 P 是单调映射，排序不变
- **(2)** Bayes 展开后先验比只依赖 $\vec{q}$，对所有候选是同一常数，排序时丢掉
- **(3)** 给定 $R$，各词出现彼此独立。这是**条件独立**（在 $R$ 下），不是无条件独立 —— `neural` 和 `network` 无条件肯定相关
- **(4)** $x_t$ 只取 0/1，乘积按"出现 / 缺失"拆；$p_t = P(x_t=1 \mid R=1)$、$u_t = P(x_t=1 \mid R=0)$
- **(5)** 不在 query 里的词假设和 relevance 无关（$q_t=0 \Rightarrow p_t=u_t$）→ 因子为 1 可丢；代数重排把缺失项归入查询常数后一并丢掉
- **(6)** 取 log，乘法变加法

结果：**文档得分 = 每个命中 query 词的 log odds ratio 之和**。这和 `tf-idf` 在倒排列表累加是同一类做法，但每项权重现在有了概率语义。

### RSJ weight + 0.5 smoothing

$p_t$、$u_t$ 怎么估？填一张 2×2 contingency table：

| | Relevant | Non-relevant | Total |
|---|---|---|---|
| 含 $t$ | $r$ | $n-r$ | $n$ |
| 不含 | $R-r$ | $(N-R)-(n-r)$ | $N-n$ |
| Total | $R$ | $N-R$ | $N$ |

MLE 直接代入 (6) 式：

$$c_t = \log \frac{r \, (N-n-R+r)}{(R-r)(n-r)}$$

小样本下分母容易为 0，$c_t = \pm \infty$ —— 某词在 20 个相关文档里碰巧没命中，就被赋 $-\infty$，含它的文档全被排死。

一个经典的修正：4 个中心 cell 各加 0.5，得到 **Robertson-Spärck Jones weight**：

$$w^{\text{RSJ}} = \log \frac{(r+0.5)(N-n-R+r+0.5)}{(R-r+0.5)(n-r+0.5)}$$

<CodeRunner lang="python" code={cell_rsj_smooth} hint="把 r 改到靠近 0 或 n，raw 公式立刻炸，smoothed 依然给出有限值" />

加 0.5 不是拍脑袋 —— 这个修正能被证明**最小化 log-odds 估计的偏差**。

### 退化到 idf

现实里很少有现成的 relevance judgments。没有 judgment，代入 $R = r = 0$ + 无偏先验 $p_t = 0.5$：

$$w^{\text{RSJ}}\big|_{R=r=0} = \log \frac{0.5 \cdot (N-n+0.5)}{0.5 \cdot (n+0.5)} = \log \frac{N-n+0.5}{n+0.5}$$

$N \gg n$ 时近似 $\log\frac{N}{n}$ —— 经典的 **inverse document frequency**。

<CodeRunner lang="python" code={cell_rsj_degen} hint="验证 R=r=0 下 RSJ 和经典 idf 的差" />

idf 最早只是一个经验直觉 —— 越稀有越有鉴别力，没有理论支撑。二十多年后，RSJ 模型在无 judgment 的极端假设下**自动退化到同一个公式**。直觉被事后追认。

这也顺便解释了为什么 idf 的各种变种（带 0.5、带 $+1$、不同偏移）在不同 collection 上表现有差：每个变种对应对 $p_t$ 和 $u_t$ 的不同先验假设。

### Ordering Principle：$(N-n)$ 从哪来

回看公式 `log(1 + (N - n + 0.5) / (n + 0.5))`，分子里那个 $(N-n)$ 有来历。

RSJ 的原始工作把 relevance weighting 做成了一个四选一问题：

**Independence Assumption**

- **I1**：词在相关文档中独立 + 在**全集合**中独立
- **I2**：词在相关文档中独立 + 在**非相关文档**中独立

**Ordering Principle**

- **O1**：文档得分只看 query 词在文档里**出现**的情况
- **O2**：文档得分**同时看出现和缺失**

组合出四个加权函数：

| | O1 | O2 |
|---|---|---|
| I1 | F1 | F3 |
| I2 | F2 | **F4** |

F4（I2 + O2）就是上面推出来的 RSJ weight。在 Cranfield 和 Keen 两个标准 IR 测试集上的完整对比：

- I1 vs I2：差异很小，独立性的具体假设不太关键
- O1 vs O2：**O2 持续优于 O1**

<CodeRunner lang="python" code={cell_f1_f4} hint="关注 common-noise 和 rare-miss 两行：O2 对这类词给出更极端的负权重" />

对"应该被惩罚的词"（高频但在相关文档里几乎不出现，典型如 stopword），O2 给出的负权重比 O1 更极端 —— 它显式利用了"缺失"这个信号。现代 BM25 里的 $(N-n+0.5)$ 就是 O2 的遗产。如果历史走岔选了 O1，BM25 的 idf 项可能只有分母，效果会差一截。

## 饱和项

代码里是这三行（暂时忽略长度部分）：

```python
num = tf[term] * (k1 + 1)
den = tf[term] + k1
score += idf * num / den
```

对应公式 $\dfrac{\text{tf} \cdot (k_1+1)}{\text{tf} + k_1}$。$\text{tf} \to \infty$ 时趋近 $k_1+1$（上限），$\text{tf}=0$ 时为 0，中间单调但**边际递减**。

<CodeRunner lang="python" code={cell_saturation} hint="同一 tf 下，k1 越大饱和越慢；同一 k1 下，tf 从 5 到 20 的增量已经很小" />

### 为什么饱和

词在文档里第 1 次出现提供"这篇在讲这个"的强信号。第 2 次加强。到第 10 次，边际贡献接近 0。BM25 设计者观察到这点，把它编码进公式。

工程效果：再怎么堆同一个词，权重也不会超过 $k_1 + 1$ 的上限。这挡住了"关键词堆砌"式 SEO —— 在 2000 年代对抗黑帽 SEO 时起过作用。

### $k_1$ 控制饱和速度

- $k_1$ 小（0.5）→ 快速饱和，第 2、3 次出现贡献都小
- $k_1$ 大（3.0）→ 慢饱和，高频词继续拉分
- $k_1 = 1.2$ 是 TREC 数据上 grid search 的经验甜点

这个饱和形式来自 **2-Poisson 模型**（词频服从双 Poisson 混合分布）的一阶近似。为什么是 $\text{tf}/(k_1+\text{tf})$ 而非 $\sqrt{\text{tf}}$ 或 $\log\text{tf}$，推导见 [2-Poisson 近似 · 饱和函数从哪来](/ir-rag/bm25/from-two-poisson-to-saturation)。

## 长度项

代码里是分母多出来的这部分：

```python
den = tf[term] + k1 * (1 - b + b * dl / avgdl)
```

`b * dl / avgdl` 是长度归一化。它嵌在饱和项分母里，和饱和 tf 共享同一个分母。

<CodeRunner lang="python" code={cell_length} hint="固定 tf=3，看 |d|/avgdl 从 0.25 到 4 时，b=0 / 0.75 / 1 三种行为" />

### 为什么嵌在分母

独立乘一个长度因子相当于对整个 score 线性缩放 —— 不区分"同样一个 tf 在长文档里是被稀释"还是"这个词本来就是分布重心"。把它嵌进分母，只缩放饱和项的"分母增量"：

- **长文档**（$\lvert d \rvert > \text{avgdl}$）：分母变大 → 同样 tf 拿到的分更少
- **短文档**（$\lvert d \rvert < \text{avgdl}$）：分母变小 → 同样 tf 拿到的分更多
- **$\lvert d \rvert = \text{avgdl}$**：$(1 - b + b \cdot 1) = 1$，退化为纯饱和项

### $b$ 控制归一化强度

- $b = 0$：完全不做长度归一化
- $b = 1$：按长度完全归一化（线性惩罚）
- $b = 0.75$：在更大规模实验上收敛到的经验值

$b < 1$ 保留了长文档的部分优势。理由：有些长文档覆盖更多内容（维基百科大词条 vs tweet），不该被简单当"啰嗦"。

## 参数与默认值

| 参数 | 默认 | 作用 |
|---|---|---|
| $k_1$ | 1.2 | 饱和速度（TREC 数据上 $k_1 \in [1.2, 2]$ 都可） |
| $b$ | 0.75 | 长度归一化强度 |
| $k_3$ | 0 或省略 | query 端饱和，query 短时用不上 |

这些数字**没有一阶原理支持**，都是实证的。Elasticsearch / Lucene 的默认就是这组，在通用文本上表现不错，但专业领域（医学 / 法律 / 代码搜索）往往要自己调。

什么时候要调、怎么调、多字段加权的 BM25F 扩展，见 [参数调优与 BM25F](/ir-rag/bm25/tuning-and-bm25f)。
